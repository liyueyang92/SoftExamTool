## 1. 文档说明

本文档基于《[软考架构师桌面工具-技术方案.md](/D:/Documents/localRep/soft/软考架构师桌面工具-技术方案.md:362)》中的爬虫模块设计，输出一份面向落地开发的专项实施计划。目标不是重复技术方案，而是把“要做什么、先做什么、在哪些文件做、每阶段怎么验收”写清楚，便于直接排期和拆任务。

## 2. 当前现状与目标差距

### 2.1 当前已具备能力

结合当前仓库代码，爬虫模块已经有一版基础实现：

  - 提供 `/crawler/test` 与 `/crawler/run` 两个接口

- Electron 主进程已存在爬虫 IPC 和运行桥接
  - [electron-app/src/main/index.ts](/D:/Documents/localRep/soft/electron-app/src/main/index.ts:749)

  - [electron-app/src/main/db/schema.ts](/D:/Documents/localRep/soft/electron-app/src/main/db/schema.ts:149)
  - 当前仅有 `crawler_rules`、`crawler_runs`
- 前端已存在规则列表、编辑弹窗、测试抓取、运行历史
  - [electron-app/src/renderer/src/views/CrawlerView.vue](/D:/Documents/localRep/soft/electron-app/src/renderer/src/views/CrawlerView.vue:1)
  - [electron-app/src/renderer/src/stores/crawler.ts](/D:/Documents/localRep/soft/electron-app/src/renderer/src/stores/crawler.ts:1)

### 2.2 当前主要缺口

- 规则模型过于简化，只覆盖单一“静态页题目列表”场景
- Python 端爬虫代码集中在单文件中，难以扩展为多适配器架构
- 不支持动态页面渲染抓取、JSON/API 抓取、Feed 导入、手动剪藏
- 不支持登录授权、会话复用、失效校验、会话元数据存储
- 无“待确认区”“人工审核后入库”的中间状态
- 抓取结果无法在入库前选择题库分组，也不支持新建分组或归档为“历年真题”
- 缺少规则版本管理、登录态管理、错误码规范、会话清理能力
- 缺少围绕 Playwright、会话文件加密、迁移脚本的专项测试

### 2.3 本次实施目标

本次开发完成后，爬虫模块需满足以下目标：

- 支持 `http_rule`、`browser_rule`、`api_json`、`feed_import`、`manual_clip` 五类适配器
- 支持规则驱动的列表页、详情页、翻页、字段抽取与字段标准化
- 支持需要登录态的站点，通过“登录并授权抓取”入口完成本地会话复用
- 支持规则测试、带登录态测试、抓取任务运行、运行历史、错误回放
- 支持去重、待确认区、人工确认后写入题库，并支持选择现有分组或新建分组
- 支持将历年真题归档到 `past_exam` 分组，并记录年份与上/下半年
- 支持会话加密存储、会话失效校验、显式清除会话、规则不带会话导出

## 3. 实施范围

### 3.1 后端范围（Python）

- 重构 `modules/crawler/` 目录，拆分适配器、规则模型、会话管理、清洗管线
- 新增登录授权接口、会话校验接口、会话删除接口
- 新增 Playwright 运行能力与 `storageState` 管理
- 新增标准错误码和统一响应约定

### 3.2 主进程范围（Electron Main）

- 扩展 IPC 通道和 Python 调用桥接
- 增加爬虫规则升级、会话元数据表、待确认区表的数据库迁移
- 增加授权窗口控制、会话导入导出限制、任务异常处理

### 3.3 前端范围（Renderer）

- 升级规则编辑界面，支持适配器、认证方式、列表/详情规则、登录判定器配置
- 增加“登录并授权抓取”“检查会话有效性”“清除登录态”入口
- 增加待确认结果预览、分组选择/新建分组和入库确认界面
- 增加更明确的运行错误提示和重试路径

### 3.4 不在本期范围

- 不支持自动破解验证码、自动填充第三方账号密码、绕过站点反爬
- 不做通用浏览器插件，不做跨设备同步登录态
- 不做大规模分布式抓取，仅支持桌面端本地单机抓取

## 4. 总体实施策略

### 4.1 演进路线

本次实施采用“**保留现有主干、逐层扩展**”的路线：

1. 保留现有 `crawler_rules + crawler_runs + test/run` 能力，确保静态规则抓取不回退。
2. 先把 Python 单文件实现拆成可扩展架构，再补多适配器。
3. 再补登录态与会话存储，因为这部分依赖规则模型升级和 Playwright 接入。
4. 最后补待确认区、审核入库、测试与发布验收。

## 5.3 Phase 3：多适配器能力落地（1.0~1.4 人周）

### 目标

补齐技术方案中定义的五类适配器，先实现 4 类自动抓取，再提供手动剪藏入口。

### 任务分解

**3.1 `http_rule`**
- 保留并增强当前能力
- 支持列表页 -> 详情页两级抓取
- 支持 `pagination.type`

**3.2 `browser_rule`**
- 引入 `Playwright`
- 支持页面等待、最终 DOM 抽取、登录态恢复后的抓取

**3.3 `api_json`**
- 允许配置接口 URL、HTTP 方法、Headers、JSON Path
- 用于解析站点公开接口或页面内嵌 JSON

**3.4 `feed_import`**
- 引入 `feedparser`
- 支持 RSS/Atom 源增量导入

**3.5 `manual_clip`**
- 支持 URL / HTML / 纯文本三种输入
- 走统一标准化与待确认流程

### 交付物

- 五类适配器接口骨架完成
- 至少四类适配器可运行
- 前端规则编辑支持选择适配器类型

### 验收标准

- 至少提供 1 个静态页、1 个动态页、1 个 API、1 个 Feed 的通路样例
- 不同适配器输出格式统一，可被同一入库流程消费
- 动态页抓取失败时可返回明确错误信息而不是静默空结果

## 5.4 Phase 4：登录授权与会话管理（1.2~1.8 人周）

### 目标

实现技术方案中的“用户手动正常登录 + 本地复用会话”链路。

- 增加授权窗口控制逻辑
- 新增 IPC：

| 待确认区 | 数据库迁移 + 主进程链路改造 | 依赖运行结果不直接入库 |
| 端到端测试 | 所有核心功能完成 | 最后统一验收 |

## 8. 风险与应对

### 8.1 Playwright 体积与打包风险

**风险：**
- Windows 打包体积上升
- 浏览器依赖下载失败或运行环境缺失

**应对：**
- 开发期先用 `playwright install chromium`
- 打包前单独验证 `browser_rule`
- 将浏览器依赖纳入安装说明或安装脚本

### 8.2 会话存储安全风险

**风险：**
- 明文会话泄露
- 规则导出误带登录态

**应对：**
- 会话文件二次加密
- 规则与会话完全分离存储
- 导出接口显式排除所有会话字段

### 8.3 规则复杂度上升带来的 UI 负担

**风险：**
- 全量配置一次性暴露，用户难以上手

**应对：**
- 默认只展示“基础模式”
- 高级字段折叠显示
- 提供站点规则模板与克隆功能

### 8.4 站点不稳定导致误判

**风险：**
- 页面结构变动导致规则失效
- 登录成功判定器不稳定

**应对：**
- 提供“测试抓取 + 预览结果”
- 保留原始响应摘要
- 会话校验使用多条件判定，不仅依赖单个选择器

## 10. 最终交付定义

本专项完成的标志不是“代码提交完成”，而是满足以下交付标准：

- 爬虫界面可创建多类型规则，并可保存、测试、运行
- 至少支持静态页、动态页、API、Feed 四类来源
- 需要登录的规则可在桌面端完成授权并复用会话
- 会话仅本地加密保存，规则导出不含登录态
- 抓取结果先进入待确认区，确认并完成分组归档后才进入题库
- 运行历史、错误提示、会话失效提示完整可见
- 模块级测试与端到端测试通过

## 11. 建议的首批开发顺序

如果立即开工，建议按以下顺序执行：

1. 先做数据库迁移与规则模型升级
2. 再拆 Python 爬虫目录并保持静态规则不回退
3. 接着落地 `browser_rule`，因为登录态能力依赖它
4. 然后实现会话存储与授权窗口链路
5. 最后补待确认区、审核入库和 UI 细节

这个顺序的原因很直接：它能尽快把高风险点 `Playwright + 登录态 + 数据迁移` 提前暴露，避免最后阶段集中返工。
