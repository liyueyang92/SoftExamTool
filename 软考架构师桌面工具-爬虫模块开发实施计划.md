本文档基于《[软考架构师桌面工具-技术方案.md](/D:/Documents/localRep/soft/软考架构师桌面工具-技术方案.md:362)》中的爬虫模块设计，输出一份面向落地开发的专项实施计划。目标不是重复技术方案，而是把“要做什么、先做什么、在哪些文件做、每阶段怎么验收”写清楚，便于直接排期和拆任务。

### 2.2 当前主要缺口

- 规则模型过于简化，只覆盖单一“静态页题目列表”场景
- Python 端爬虫代码集中在单文件中，难以扩展为多适配器架构
- 不支持动态页面渲染抓取、JSON/API 抓取、Feed 导入、手动剪藏
- 不支持登录授权、会话复用、失效校验、会话元数据存储
- 无“待确认区”“人工审核后入库”的中间状态
- 抓取结果无法在入库前选择题库分组，也不支持新建分组或归档为“历年真题”
- 缺少规则版本管理、登录态管理、错误码规范、会话清理能力
- 规则配置仍依赖手工填写 CSS Selector / XPath，对非技术用户不友好，缺少页面点选、字段映射和实时预览能力
- 缺少围绕 Playwright、会话文件加密、迁移脚本的专项测试

### 2.3 本次实施目标

本次开发完成后，爬虫模块需满足以下目标：

- 支持 `http_rule`、`browser_rule`、`api_json`、`feed_import`、`manual_clip` 五类适配器
- 支持规则驱动的列表页、详情页、翻页、字段抽取与字段标准化
- 支持通过可视化规则配置器点选页面内容，辅助生成列表、详情、分页和字段选择器
- 支持需要登录态的站点，通过“登录并授权抓取”入口完成本地会话复用
- 支持规则测试、带登录态测试、抓取任务运行、运行历史、错误回放
- 支持去重、待确认区、人工确认后写入题库，并支持选择现有分组或新建分组
- 支持将历年真题归档到 `past_exam` 分组，并记录年份与上/下半年
- 支持会话加密存储、会话失效校验、显式清除会话、规则不带会话导出

## 3. 实施范围

### 3.1 后端范围（Python）

- 重构 `modules/crawler/` 目录，拆分适配器、规则模型、会话管理、清洗管线
- 新增登录授权接口、会话校验接口、会话删除接口
- 新增 Playwright 运行能力与 `storageState` 管理
- 新增页面检查接口，用于返回页面快照、DOM 节点路径、候选选择器和字段预览结果
- 新增标准错误码和统一响应约定

### 3.2 主进程范围（Electron Main）

- 扩展 IPC 通道和 Python 调用桥接
- 增加爬虫规则升级、会话元数据表、待确认区表的数据库迁移
- 增加授权窗口控制、可视化检查窗口控制、会话导入导出限制、任务异常处理

### 3.3 前端范围（Renderer）

- 升级规则编辑界面，支持适配器、认证方式、列表/详情规则、登录判定器配置
- 增加可视化规则配置器，支持页面预览、节点点选、字段映射、选择器生成和抽取结果预览
- 增加“登录并授权抓取”“检查会话有效性”“清除登录态”入口
- 增加待确认结果预览、分组选择/新建分组和入库确认界面
- 增加更明确的运行错误提示和重试路径

### 3.4 不在本期范围

- 不支持自动破解验证码、自动填充第三方账号密码、绕过站点反爬
- 不做通用浏览器插件，不做跨设备同步登录态
- 可视化配置器只作为规则生成辅助，不承诺对所有复杂站点自动识别；复杂场景仍保留高级 JSON / Selector 手工编辑
- 不做大规模分布式抓取，仅支持桌面端本地单机抓取

## 4. 总体实施策略

### 4.1 演进路线

本次实施采用“**保留现有主干、逐层扩展**”的路线：

1. 保留现有 `crawler_rules + crawler_runs + test/run` 能力，确保静态规则抓取不回退。
2. 先把 Python 单文件实现拆成可扩展架构，再补多适配器。
3. 再补登录态与会话存储，因为这部分依赖规则模型升级和 Playwright 接入。
4. 最后补待确认区、审核入库、测试与发布验收。

## 5.3 Phase 3：多适配器能力落地（1.0~1.4 人周）

### 目标

补齐技术方案中定义的五类适配器，先实现 4 类自动抓取，再提供手动剪藏入口。

### 任务分解

**3.1 `http_rule`**
- 保留并增强当前能力
- 支持列表页 -> 详情页两级抓取
- 支持 `pagination.type`
- 支持 `canonical_url` 提取

**3.2 `browser_rule`**
- 引入 `Playwright`
- 支持页面等待、最终 DOM 抽取、登录态恢复后的抓取
- 支持基础页面超时与截图调试

**3.3 `api_json`**
- 允许配置接口 URL、HTTP 方法、Headers、JSON Path
- 用于解析站点公开接口或页面内嵌 JSON

**3.4 `feed_import`**
- 引入 `feedparser`
- 支持 RSS/Atom 源增量导入

**3.5 `manual_clip`**
- 支持 URL / HTML / 纯文本三种输入
- 走统一标准化与待确认流程

### 交付物

- 五类适配器接口骨架完成
- 至少四类适配器可运行
- 前端规则编辑支持选择适配器类型

### 验收标准

- 至少提供 1 个静态页、1 个动态页、1 个 API、1 个 Feed 的通路样例
- 不同适配器输出格式统一，可被同一入库流程消费
- 动态页抓取失败时可返回明确错误信息而不是静默空结果

## 5.4 Phase 4：登录授权与会话管理（1.2~1.8 人周）

### 目标

实现技术方案中的“用户手动正常登录 + 本地复用会话”链路。

### 任务分解

**主进程：**
- 增加授权窗口控制逻辑
- 新增 IPC：
  - `crawler:authStart`
  - `crawler:listSessions`
  - `crawler:validateSession`
  - `crawler:deleteSession`
- 授权窗口关闭、异常关闭、超时关闭的兜底处理

**Python 端：**
- 新增接口：
  - `POST /crawler/auth/start`
  - `GET /crawler/auth/{siteId}/sessions`
  - `POST /crawler/auth/{siteId}/validate`
  - `DELETE /crawler/auth/{siteId}/sessions/{accountAlias}`
- 基于 `Playwright persistent context` 采集 `storageState`
- 引入会话加密存储：
  - `auth/state_crypto.py`
  - `auth/session_store.py`
- 支持 `storageState -> CookieJar` 转换

**前端：**
- 在 `CrawlerView.vue` 中新增：
  - “登录并授权抓取”
  - “检查会话有效性”
  - “清除登录态”
  - `account_alias` 选择
- 授权完成后能马上执行“带登录态测试抓取”

**错误码：**
- 统一支持：
  - `CRAWLER_AUTH_REQUIRED`
  - `CRAWLER_AUTH_EXPIRED`
  - `CRAWLER_AUTH_INVALID`
  - `CRAWLER_AUTH_REVOKED`
  - `CRAWLER_LOGIN_TIMEOUT`

### 交付物

- 站点授权登录入口可用
- 会话文件本地加密存储可用
- 抓取任务可显式选择 `account_alias`

### 验收标准

- 用户可对一个需要登录的规则完成登录授权并成功测试抓取
- 会话过期时能收到正确错误码与重新登录提示
- 清除会话后，后续抓取不能继续复用旧状态
- 规则导出结果不包含任何登录态数据

## 5.4.5 Phase 4.5：可视化规则配置器（0.5~0.8 人周）

### 目标

在保留高级 JSON / Selector 手工编辑能力的基础上，提供“页面预览 -> 点击选择 -> 字段映射 -> 实时预览 -> 写回规则”的可视化配置链路，降低配置爬虫规则的门槛。

该阶段不引入新的抓取结果模型，最终仍输出标准 `rule_json`，由 Phase 3 的适配器和 Phase 5 的待确认区继续消费。

### 任务分解

**Python 端：**
- 新增 `inspector/page_loader.py`，根据规则草稿加载页面 HTML 或 Playwright 渲染后的 DOM
- 新增 `inspector/selector_generator.py`，根据点选节点生成候选 CSS Selector / XPath
- 新增 `inspector/preview.py`，使用当前规则草稿执行局部抽取并返回预览结果
- 新增页面检查接口：
  - `POST /crawler/inspect/load`
  - `POST /crawler/inspect/preview`
  - `POST /crawler/inspect/suggest-selector`
- 支持带登录态的页面检查，复用 Phase 4 的 `account_alias` 与 `storageState`
- 返回页面快照时过滤脚本执行能力，避免在前端预览中直接运行第三方页面脚本

**主进程：**
- 新增 IPC：
  - `crawler:inspectLoad`
  - `crawler:inspectPreview`
  - `crawler:suggestSelector`
- 将可视化检查请求转发到 Python 服务，并统一处理超时、页面过大、登录态缺失等错误
- 限制页面快照缓存生命周期，避免长期保存第三方页面内容

**前端：**
- 在规则编辑弹窗中增加“可视化配置”标签页，与“高级配置”并存
- 新增 `VisualRuleBuilder.vue`，负责组织页面预览、字段面板和预览结果
- 新增 `SelectorPicker.vue`，支持 hover 高亮、点击选择节点、查看候选选择器
- 新增 `ExtractionPreview.vue`，展示当前规则能抽取到的列表项和字段
- 支持点选并写回以下配置：
  - 列表项选择器
  - 详情链接选择器
  - 标题、正文、选项、答案、解析、标签等字段选择器
  - 下一页按钮或分页链接选择器
  - 登录成功判定选择器
- 支持“使用该选择器”“换一个候选”“手工微调”三种操作
- 可视化配置生成的内容必须同步写入 `rule_json`，保存后无需依赖可视化组件即可运行

### 交付物

- 用户可在规则编辑界面通过点选页面元素生成基础抓取规则
- 可视化点选生成的规则可保存、测试、运行
- 同一条规则可在“可视化配置”和“高级配置”之间切换查看
- 至少支持 `http_rule` 和 `browser_rule` 两类页面的可视化配置

### 验收标准

- 对一个静态列表页，用户不手写 Selector 也能完成列表项、标题、详情链接配置并成功测试抓取
- 对一个动态页面，用户可在 Playwright 渲染后的页面快照中点选字段并完成预览

## 8. 风险与应对

### 8.1 Playwright 体积与打包风险

**风险：**
- Windows 打包体积上升
- 浏览器依赖下载失败或运行环境缺失

**应对：**
- 开发期先用 `playwright install chromium`
- 打包前单独验证 `browser_rule`
- 将浏览器依赖纳入安装说明或安装脚本

### 8.2 会话存储安全风险

**风险：**
- 明文会话泄露
- 规则导出误带登录态

**应对：**
- 会话文件二次加密
- 规则与会话完全分离存储
- 导出接口显式排除所有会话字段

### 8.3 规则复杂度上升带来的 UI 负担

**风险：**
- 全量配置一次性暴露，用户难以上手
- 可视化点选和高级配置并存后，界面入口过多

**应对：**
- 默认只展示“基础模式”
- 高级字段折叠显示
- 提供站点规则模板与克隆功能
- 将可视化配置作为默认入口，高级 JSON / Selector 编辑作为兜底入口

### 8.4 站点不稳定导致误判

**风险：**
- 页面结构变动导致规则失效
- 登录成功判定器不稳定
- 可视化生成的选择器过于脆弱，页面轻微改版后匹配为空或匹配过多

**应对：**
- 提供“测试抓取 + 预览结果”
- 保留原始响应摘要
- 会话校验使用多条件判定，不仅依赖单个选择器
- 选择器生成优先使用稳定属性、文本结构和层级约束，并在预览中提示匹配数量

### 8.5 页面快照与第三方内容安全风险

**风险：**
- 预览页面中直接运行第三方脚本带来安全风险
- 页面快照体积过大，影响桌面端内存和渲染性能
- 登录态页面快照可能包含敏感信息

**应对：**
- 前端预览使用清洗后的静态快照，不直接执行第三方页面脚本
- 对快照大小、资源加载和缓存生命周期设置上限
- 带登录态检查只在本地会话中执行，快照不长期持久化，不随规则导出

## 11. 建议的首批开发顺序

如果立即开工，建议按以下顺序执行：

1. 先做数据库迁移与规则模型升级
