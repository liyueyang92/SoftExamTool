# 软考系统架构设计师考试辅助桌面工具 开发实施计划

## 1. 文档说明

本文档是《软考架构师桌面工具-技术方案.md》的配套实施指南，描述从零到可交付安装包的完整开发路径，包括各阶段目标、任务分解、依赖关系、验收标准及已知风险点。

**总体原则：**
- 优先打通技术主干（IPC 通信、Python 服务管理、数据库），再叠加业务功能。
- 每个阶段结束时必须有可运行的交付物，而非半完成的代码。
- 高风险技术点（SQLCipher 编译、PyInstaller 打包）在第一阶段提前验证，不留到末期。

**总工期参考：** 约 22~28 人周（单人全职），可按模块并行缩短。

## 3. Phase 0：环境搭建与项目骨架（1~2 人周）

### 3.1 目标

两进程（Electron 主进程 + Python FastAPI）能够互相通信并在开发模式下一键启动，证明技术主干可行。本阶段交付物是一个**可运行的最小壳**，不含任何业务逻辑，但所有后续阶段依赖的通信机制、构建链路、目录约定均在此确定。

### 3.5 关键配置文件实现

**`python-service/config.py`** — 路径适配与环境变量读取：
```python
import os
import sys
from pathlib import Path

def get_resource_path(relative_path: str) -> Path:
    """兼容开发模式与 PyInstaller 打包后的资源路径"""
    if hasattr(sys, '_MEIPASS'):
        base = Path(sys._MEIPASS)
    else:
        base = Path(__file__).parent
    return base / relative_path

class Settings:
    port: int = int(os.environ.get('PYTHON_PORT', 8765))
    token: str = os.environ.get('INTERNAL_TOKEN', '')
    log_level: str = os.environ.get('LOG_LEVEL', 'INFO')
    # 用户数据目录由主进程通过环境变量传入
    data_dir: Path = Path(os.environ.get('APP_DATA_DIR', Path.home() / '.soft-exam-tool'))

settings = Settings()
```

**`python-service/middleware/auth.py`** — Token 认证中间件：
```python
from fastapi import Request, HTTPException
from fastapi.responses import JSONResponse
from starlette.middleware.base import BaseHTTPMiddleware
from config import settings

class TokenAuthMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):
        # /health 端点允许无 Token 访问（主进程健康检查用）
        if request.url.path == '/health':
            return await call_next(request)
        token = request.headers.get('X-Internal-Token', '')
        if token != settings.token or not settings.token:
            return JSONResponse({'detail': 'Unauthorized'}, status_code=403)
        return await call_next(request)
```

**`python-service/main.py`** — 最小 FastAPI 应用：
```python
from fastapi import FastAPI
from loguru import logger
from config import settings
from middleware.auth import TokenAuthMiddleware

app = FastAPI(title='SoftExam Python Service', docs_url=None, redoc_url=None)
app.add_middleware(TokenAuthMiddleware)

@app.get('/health')
async def health():
    return {'status': 'ok', 'version': '0.1.0'}

@app.get('/ping')
async def ping():
    return {'message': 'pong'}

if __name__ == '__main__':
    import uvicorn
    logger.info(f'Starting service on port {settings.port}')
    uvicorn.run(app, host='127.0.0.1', port=settings.port, log_level=settings.log_level.lower())
```

### 3.6 IPC 通信打通

实现最小 IPC 闭环，验证完整通信链路：

**通信链路：** 渲染进程 → preload `contextBridge` → 主进程 IPC → HTTP → Python → 返回

**Step 1：主进程 Python 进程管理（`python-manager.ts`）**

```typescript
import { ChildProcess, spawn } from 'child_process'
import { app } from 'electron'
import path from 'path'
import net from 'net'

let pythonProcess: ChildProcess | null = null

// 找一个可用的随机端口
async function getFreePort(): Promise<number> {
  return new Promise((resolve) => {
    const srv = net.createServer()
    srv.listen(0, '127.0.0.1', () => {
      const port = (srv.address() as net.AddressInfo).port
      srv.close(() => resolve(port))
    })
  })
}

export async function startPythonService(): Promise<{ port: number; token: string }> {
  const port = await getFreePort()
  const token = require('crypto').randomBytes(32).toString('hex')

  // 开发模式用 python，打包后用 resources/python-service/python-service.exe
  const isDev = !app.isPackaged
  const pythonCmd = isDev
    ? path.join(__dirname, '../../python-service/.venv/Scripts/python.exe')
    : path.join(process.resourcesPath, 'python-service', 'python-service.exe')
  const scriptArg = isDev ? [path.join(__dirname, '../../python-service/main.py')] : []

  pythonProcess = spawn(pythonCmd, scriptArg, {
    env: {
      ...process.env,
      PYTHON_PORT: String(port),
      INTERNAL_TOKEN: token,
      APP_DATA_DIR: app.getPath('userData'),
      LOG_LEVEL: isDev ? 'DEBUG' : 'INFO'
    },
    stdio: ['ignore', 'pipe', 'pipe']
  })

  pythonProcess.stdout?.on('data', (d) => console.log('[Python]', d.toString().trim()))
  pythonProcess.stderr?.on('data', (d) => console.error('[Python ERR]', d.toString().trim()))
  pythonProcess.on('exit', (code) => console.warn(`[Python] exited with code ${code}`))

  // 轮询健康检查，最多等待 15 秒
  await waitForService(port)
  return { port, token }
}

async function waitForService(port: number, maxRetries = 30): Promise<void> {
  for (let i = 0; i < maxRetries; i++) {
    try {
      const res = await fetch(`http://127.0.0.1:${port}/health`)
      if (res.ok) return
    } catch {
      // 服务尚未就绪
    }
    await new Promise((r) => setTimeout(r, 500))
  }
  throw new Error('Python service failed to start within 15 seconds')
}

export function stopPythonService(): void {
  if (pythonProcess && !pythonProcess.killed) {
    pythonProcess.kill()
    pythonProcess = null
  }
}
```

**Step 2：主进程入口（`src/main/index.ts`）**关键部分：

```typescript
import { app, BrowserWindow, ipcMain } from 'electron'
import { startPythonService, stopPythonService } from './python-manager'

let pythonPort: number
let pythonToken: string

app.whenReady().then(async () => {
  const win = createWindow()

  try {
    const svc = await startPythonService()
    pythonPort = svc.port
    pythonToken = svc.token
    // 通知渲染进程服务已就绪
    win.webContents.send('python:ready')
  } catch (e) {
    win.webContents.send('python:error', (e as Error).message)
  }

  // Phase 0 调试：ping 通道
  ipcMain.handle('debug:ping', async () => {
    const res = await fetch(`http://127.0.0.1:${pythonPort}/ping`, {
      headers: { 'X-Internal-Token': pythonToken }
    })
    return res.json()
  })
})

app.on('window-all-closed', () => {
  stopPythonService()
  if (process.platform !== 'darwin') app.quit()
})
```

**Step 3：Preload 脚本（`src/preload/index.ts`）**：

```typescript
import { contextBridge, ipcRenderer } from 'electron'

contextBridge.exposeInMainWorld('electronAPI', {
  // Phase 0 调试接口
  ping: () => ipcRenderer.invoke('debug:ping'),
  onPythonReady: (cb: () => void) => ipcRenderer.on('python:ready', cb),
  onPythonError: (cb: (msg: string) => void) =>
    ipcRenderer.on('python:error', (_e, msg) => cb(msg)),
})

// 类型声明（放入 src/renderer/src/env.d.ts）
export interface ElectronAPI {
  ping: () => Promise<{ message: string }>
  onPythonReady: (cb: () => void) => void
  onPythonError: (cb: (msg: string) => void) => void
}

declare global {
  interface Window { electronAPI: ElectronAPI }
}
```

### 3.7 开发启动脚本

**`scripts/dev-start.ps1`** — 一键启动两个进程，合并日志，`Ctrl+C` 同时停止：

```powershell
# 需在仓库根目录执行：.\scripts\dev-start.ps1
$ErrorActionPreference = 'Stop'

# 启动 Python 服务（使用虚拟环境）
$pythonProc = Start-Process -FilePath ".\python-service\.venv\Scripts\python.exe" `
  -ArgumentList ".\python-service\main.py" `
  -Environment @{ PYTHON_PORT='8765'; INTERNAL_TOKEN='dev-token'; LOG_LEVEL='DEBUG' } `
  -PassThru -NoNewWindow

Write-Host "[Dev] Python service PID: $($pythonProc.Id)" -ForegroundColor Cyan

# 启动 Electron + Vite 开发服务器
Set-Location electron-app
try {
  pnpm run dev
} finally {
  # Electron 退出后，关闭 Python 进程
  if (-not $pythonProc.HasExited) {
    Stop-Process -Id $pythonProc.Id -Force
    Write-Host "[Dev] Python service stopped." -ForegroundColor Yellow
  }
  Set-Location ..
}
```

### 3.8 常见问题排查

| 问题 | 现象 | 排查步骤 |
|------|------|----------|
| SQLCipher 编译失败 | `gyp ERR!` 或 `MSVS not found` | 1. 确认 VS Build Tools 2022 已装 C++ 组件；2. 执行 `npm config set msvs_version 2022`；3. 以管理员权限重新执行 `electron-rebuild` |
| Python 进程启动后立即退出 | 日志显示 `exited with code 1` | 在终端手动运行 `python python-service/main.py` 查看 Python 报错；常见原因：缺少依赖（运行 `pip install -r requirements.txt`）、端口被占用 |
| 主进程 `fetch` 请求到 Python 超时 | `waitForService` 抛出 15s 超时异常 | 检查 Python 进程是否正常启动；`PYTHON_PORT` 是否与 Python 实际监听端口一致；防火墙是否拦截本地回环请求 |
| 渲染进程 `window.electronAPI` 为 `undefined` | `TypeError: Cannot read properties of undefined` | 确认 `electron.vite.config.ts` 中 `preload` 路径正确；`webPreferences` 中 `contextIsolation: true` 且 `preload` 字段已指向编译后的 preload 文件 |
| `node_modules` 中 Electron 版本与 `@journeyapps/sqlcipher` 不匹配 | 运行时 `Error: The module ... was compiled against a different Node.js version` | 重新执行 `pnpm exec electron-rebuild -f -w @journeyapps/sqlcipher`；确保 `package.json` 中 `electron` 版本与 rebuild 时一致 |
| Python 服务返回 403 | 控制台报 `Unauthorized` | 检查主进程传给 Python 的 `INTERNAL_TOKEN` 与请求 Header 中的 `X-Internal-Token` 值是否一致；开发模式下确认 `.env` 配置已加载 |

## 4. Phase 1：基础设施层（2~3 人周）

### 4.1 目标

完成所有业务功能共用的底层能力：数据库初始化与迁移、规范化 IPC、任务状态机、WebSocket 进度通道。此阶段不实现任何业务功能，但为后续所有阶段打地基。

### 4.2 数据库初始化与迁移

**迁移机制：**
- 在 `src/main/db/` 下创建迁移脚本目录，每个迁移文件命名为 `0001_init.sql`、`0002_add_tasks.sql` 等。
- 主进程启动时读取 `PRAGMA user_version`，按序执行未执行的迁移文件，完成后更新版本号。

**本阶段需建立的表：**
按技术方案 5.1.2 节执行，至少建立：`questions`、`answer_records`、`study_plans`、`tasks`、`documents`、`doc_chunks`，以及 FTS5 虚拟表与同步触发器。

**加密初始化：**
- 首次启动时检测数据库是否存在。
- 若不存在，生成随机 32 字节密钥，通过 `keytar` 存入 Windows Credential Manager，用该密钥初始化 SQLCipher 数据库。
- 后续启动时从 Credential Manager 读取密钥并打开数据库。

### 4.3 IPC 规范层

**`ipc-channels.ts`** — 注册所有通道名常量：
```typescript
export const IPC = {
  DB_QUERY_QUESTIONS: 'db:queryQuestions',
  DB_INSERT_QUESTION: 'db:insertQuestion',
  AI_GENERATE_PAPER:  'ai:generatePaper',
  TASK_CANCEL:        'task:cancel',
  // ... 其余通道
} as const
```

**`ipc-handler.ts`** — 封装统一的注册与响应格式：
```typescript
type IpcResponse<T> = { success: true; data: T } | { success: false; error: { code: string; message: string } }

function registerHandler<T>(channel: string, fn: (args: unknown) => Promise<T>) {
  ipcMain.handle(channel, async (_, args) => {
    try {
      return { success: true, data: await fn(args) }
    } catch (e) {
      return { success: false, error: { code: e.code ?? 'UNKNOWN', message: e.message } }
    }
  })
}
```

**渲染进程封装层** — 所有调用带超时（默认 30 秒，AI 任务 120 秒），超时后自动发起取消。

### 4.4 任务状态机

在 `src/main/task-manager.ts` 中实现：
- `createTask(type, payload)` → 写入 `tasks` 表，返回 `taskId`（UUID）
- `updateTask(taskId, status, result?)` → 更新状态
- `getTask(taskId)` → 查询状态
- 主进程启动时扫描所有 `running` 状态任务，批量标记为 `failed`

### 4.5 WebSocket 进度通道

Python 端在 FastAPI 中注册 `/ws/progress/{task_id}` WebSocket 端点：
- 耗时任务执行期间定期推送 `{ taskId, progress, message }` 消息。
- 任务完成或失败时推送最终状态消息并关闭连接。

Electron 主进程建立 WebSocket 客户端连接，收到消息后通过 IPC 转发给渲染进程，实现断线指数退避重连（1s → 2s → 4s → 最大 30s）。

### 4.6 全局布局与路由骨架

搭建主界面框架，不实现功能页面内容，仅确保导航、布局、暗黑模式切换正常工作：
- 左侧导航栏（图标 + 文字）
- 内容区路由占位页面
- 顶部状态栏（Python 服务状态指示器、倒计时显示区）
- Pinia Store 骨架（各 Store 创建但不含业务逻辑）

### 4.7 验收标准

- [ ] 数据库迁移脚本在全新安装和升级场景下均正常运行
- [ ] SQLCipher 密钥从 Credential Manager 读写正常
- [ ] 所有 IPC 调用超时后返回标准错误格式
- [ ] 任务状态机在模拟进程重启后正确标记残留任务为 failed
- [ ] WebSocket 进度消息从 Python → 主进程 → 渲染进程链路联通
- [ ] 全局导航与暗黑模式切换正常

## 6. Phase 3：AI 与 PDF 集成（3~4 人周）

### 6.1 目标

完成 PDF 导入与知识点归类、AI 智能出题、AI 主观题评分三个核心 AI 功能，同时完成 AI 配置管理（远程 API / 本地 Ollama 切换）。

### 6.2 AI 配置管理

**设置页 → AI 配置面板：**
- 模式切换：远程 API / 本地 Ollama
- 远程 API：厂商选择（OpenAI、通义千问、文心一言等 OpenAI 兼容接口）、Base URL、API Key、模型名称
- 本地 Ollama：地址（默认 `http://localhost:11434`）、模型名称
- 测试连接按钮：发送简单 prompt，验证配置有效
- API Key 通过 `keytar` 存储，界面仅显示掩码

**Python 端 AI 提供者抽象：**
```python
class AIProvider(Protocol):
    async def chat(self, messages: list[dict], stream: bool = False) -> AsyncIterator[str]: ...

class OllamaProvider(AIProvider): ...
class OpenAICompatProvider(AIProvider): ...
```
根据主进程传入的配置参数初始化对应提供者。

### 6.3 PDF 导入与解析

**导入流程：**
1. 用户选择 PDF 文件，主进程读取文件路径，调用 `task:createPdfTask` 创建任务。
2. 主进程向 Python 服务发送解析请求（传入文件路径与任务 ID）。
3. Python 端检查 MD5 缓存，命中则直接返回；未命中则解析。
4. 解析进度通过 WebSocket 推送（每处理 10 页推送一次百分比）。
5. 完成后结果写入 `documents` + `doc_chunks` 表，任务状态更新为 `completed`。

**OCR 触发条件：** 某页提取到文字少于 50 个字符，则视为扫描页，转入 OCR 流程；若 Tesseract 语言包未安装，提示用户下载后重试。

**知识点归类（阶段一，简单版）：**
- 使用正则匹配章节标题（如"第X章"、"X.X"格式）进行段落层次划分。
- 基于关键词字典对段落打上知识点标签（不依赖向量模型，向量归类在 Phase 5 实现）。

**文档库页面：** 展示已导入文档列表（标题、页数、导入时间、知识点分布），支持全文检索，点击查看原文段落。

### 6.4 AI 智能出题

**出题配置面板：**
- 知识点范围（多选），难度等级，题型组合（单选/多选/案例/论文各几题），命题倾向（高频考点/技术趋势可勾选）
- 可基于已导入文档内容出题，或纯基于模型知识

**出题流程：**
1. 渲染进程提交参数，主进程创建 `ai_generate` 任务。
2. 调用 Python AI 模块，构建约束性 System Prompt（含题型格式要求、JSON Schema 约束）。
3. 模型输出流式返回，Python 端聚合完整响应后通过 pydantic 校验结构，最多重试 3 次。
4. 校验通过后批量插入 `questions` 表（`source_type = 'ai_generated'`），返回生成的题目列表。
5. 前端展示预览，用户可编辑后确认保存或直接丢弃。

**一键生成整卷：** 按真实考试结构（综合知识 75 题 + 案例分析 2 题 + 论文 1 题）自动配置参数并批量调用出题接口。

### 6.5 AI 主观题评分

**评分请求参数：** 题目内容、参考答案要点、评分维度细则（内置论文评分维度配置文件）、用户答案。

**评分流程：**
1. 用户提交答案后，前端展示"AI 评分中"进度状态。
2. Python AI 模块调用模型，输出 JSON 格式：`{ total_score, dimension_scores: [...], feedback, suggestions }`。
3. 结果写入 `answer_records.ai_feedback`，前端展示维度雷达图与改进建议。
4. 用户可申请"人工复核"，进入手动填分模式，手动分数写入 `manual_override_score`。

### 6.6 验收标准

- [ ] 导入一份 200 页以上的 PDF，进度条正常推进，解析完成后文档库可查看
- [ ] 同一 PDF 重复导入，命中缓存，不重复解析
- [ ] 配置 AI（远程或本地），测试连接成功
- [ ] AI 生成 10 道单选题，格式正确，可保存至题库
- [ ] 提交一篇论文，AI 评分返回各维度得分与评语
- [ ] Python 服务重启后，前端能正确展示服务不可用提示，不崩溃

## 8. Phase 5：爬虫与高级功能（3~4 人周）

### 8.1 目标

### 8.2 爬虫引擎

**规则管理页面：**
- 列出所有站点规则（内置 + 用户自定义），每条显示站点名称、最后抓取时间、抓取题数。

**抓取执行：**
- 用户手动触发（单个规则或全部规则），前端展示进度与日志。
- Python 爬虫模块执行：随机延迟（500ms~3000ms）、User-Agent 池、可选 robots.txt 检查。
- 抓取结果清洗（去除广告/排版错误）、去重（基于题目内容哈希），入库时标记 `source_type = 'crawled'`。

**手动剪藏：** 提供系统级悬浮工具栏（Electron 透明窗口），用户在浏览器中划选文字后点击"保存到题库"，弹出录入对话框预填内容。

### 8.3 向量检索与语义归类

**依赖：** Python `chromadb` + `sentence-transformers`（`m3e-small` 模型，约 100MB，首次使用时下载）。

**索引构建：**
- PDF 导入完成后，后台异步任务对 `doc_chunks` 逐批生成嵌入向量（每批 32 条），写入 chromadb collection，`vector_id` 回写 SQLite。
- 进度通过 WebSocket 推送，不阻塞其他操作。

**知识点语义归类（替代 Phase 3 的关键词版本）：**
- 将官方大纲各考点标签预先向量化，建立"大纲向量库"。
- 对每个 `doc_chunk` 计算与大纲标签的余弦相似度，取 Top-3 标签写入 `knowledge_tag`。
- 前端提供批量校正界面，用户可修改归类结果。

**AI 问答 RAG：**
- AI 对话页面中，用户提问时先在 chromadb 中检索 Top-5 相关段落，作为上下文注入 System Prompt，回答中标注来源文档与页码。

### 8.4 知识图谱

**构建逻辑（Python 端）：**
- 批量计算 `doc_chunks` 段落间嵌入相似度，提取相似度 > 阈值（默认 0.8）的关联对。
- 以知识点标签为节点，关联强度为边权重，生成图数据（`{ nodes: [...], edges: [...] }`）。

**前端可视化：**
- 使用 ECharts `graph` 类型渲染，支持拖拽、缩放、点击节点跳转到相关题目或文档段落。
- 节点大小表示关联题目数量，节点颜色表示用户对该知识点的掌握程度（正确率映射）。
- 节点超过 200 个时，布局计算迁移至 WebWorker 避免 UI 卡顿。

### 8.5 论文专项训练

**结构引导编辑器（前端实现为主）：**
- 分段编辑区：摘要（300字）、背景与挑战（500字）、技术方案（1500字）、项目实践（1000字）、总结（300字）。
- 每段实时字数统计，超/未达建议字数时高亮提示。
- 每段右侧展示 AI 写作建议（按需触发，不自动调用）。

**论文素材库：**
- 用户录入条目：项目名称、背景描述、技术难点、解决方案、项目成果（结构化表单）。
- 写作时可通过侧边栏语义检索素材，一键插入引用。
- 从导入文档中标记的段落可批量导入素材库。

**论文历史记录：** 每次保存自动生成版本快照，支持版本对比与回滚。

### 8.6 验收标准

- [ ] 使用内置规则成功抓取目标站点题目，去重正确
- [ ] 导入 200 页 PDF 后，向量索引后台构建完成，不影响前台操作


**触发逻辑：** 在 `useAchievementStore` 中监听做题、打卡等事件，检查未解锁成就条件，满足时写入 `achievements` 表并推送桌面通知 + 界面动画。

**成就展示页：** 徽章墙，未解锁显示轮廓，已解锁显示彩色图标与解锁时间。

### 9.4 健康学习提醒

- 通过 `study_sessions` 表记录每次连续学习开始时间。
- 主进程每 5 分钟检查一次：若当前连续学习时长 ≥ 45 分钟，弹出悬浮卡片提示（Electron 透明置顶窗口），提示活动后可关闭。
- 设置页可调整提醒阈值（30 / 45 / 60 分钟）或关闭。
- 每日学习时长统计展示在仪表盘，建议合理范围以文字注明。

### 9.5 数据备份与恢复

**备份功能：**
- 手动备份：用户选择目标目录，Python 备份模块打包加密压缩。
- 定时自动备份：每天首次启动时检查，超过 24 小时未备份则自动执行，最多保留 7 份。
- WebDAV 备份：用户在设置中填写 WebDAV 地址/账号，测试连接后启用，备份完成自动上传。

**恢复功能：**
- 选择本地备份文件或从 WebDAV 下载，输入加密密码，解压后替换数据目录。
- 恢复完成提示重启应用，重启后重新初始化数据库连接。

**注意：** 备份操作会暂时锁定数据库（WAL checkpoint），需提示用户等待完成。

### 9.6 验收标准

- [ ] 番茄钟 25 分钟工作时段结束后正确推送通知，时长记录到统计
- [ ] 完成 100 道题后"破冰"成就自动解锁，界面弹出动画
- [ ] 连续学习 45 分钟后健康提醒卡片弹出，关闭后计时重置
- [ ] 手动备份生成加密 ZIP 文件，文件可用恢复功能还原，数据完整
- [ ] WebDAV 备份上传成功，从 WebDAV 下载恢复流程完整

## 10. Phase 7：集成打包与发布（2~3 人周）

### 10.1 目标

输出可分发给用户的签名安装包，完成端到端功能验证。

### 10.2 PyInstaller 打包

**打包脚本** `scripts/build-python.ps1`：
```powershell
pyinstaller --name python-service `
  --onedir `
  --noconsole `
  --add-data "modules/crawler/rules;modules/crawler/rules" `
  --add-data "modules/ai/scoring_rubrics;modules/ai/scoring_rubrics" `
  main.py
```

**验证项：**
- 打包后的可执行文件在未安装 Python 的干净 VM 上能正常启动。
- `sys._MEIPASS` 路径适配正确，资源文件可访问。
- 启动时间记录：目标 < 10 秒（若超过需分析哪些库导致膨胀）。

**OCR 语言包单独处理：** 不打入主包，首次 OCR 时提示用户下载 `chi_sim.traineddata` 至用户数据目录。

### 10.3 Electron Builder 配置

**`electron-builder.yml` 关键配置：**
```yaml
extraResources:
  - from: resources/python-service/
    to: python-service/
win:
  target: [nsis, portable]
  certificateFile: ${env.CERT_FILE}
  certificatePassword: ${env.CERT_PASSWORD}
  signingHashAlgorithms: [sha256]
nsis:
  oneClick: false
  allowToChangeInstallationDirectory: true
  installerLanguages: [zh_CN]
```

**代码签名：**
- 证书文件路径与密码通过环境变量注入，不写入代码仓库。
- 打包完成后运行 `signtool verify /pa /v installer.exe` 验证签名有效性。
- 若无 EV 证书，至少使用标准 OV 证书签名，否则记录 SmartScreen 警告已知问题。

### 10.4 自动更新配置

**`electron-updater` 配置：**
- 更新服务器：GitHub Releases（`publish: github`）。
- 应用启动 10 秒后静默检查更新，有新版本时界面底部展示更新提示（不强制安装）。
- 更新包中包含新版 Python 可执行文件夹，主进程更新完成后替换 `resources/python-service/` 目录，确保 Python 服务版本与应用版本一致。

### 10.5 端到端测试清单

使用 Playwright 自动化覆盖以下核心场景，每次发布前必须全部通过：

| 场景 | 测试步骤 |
|------|----------|
| 首次安装启动 | 安装 → 启动 → Python 服务初始化完成 → 首页正常显示 |
| PDF 导入 | 选择测试 PDF → 导入 → 进度完成 → 文档库出现条目 |
| AI 出题 | 配置 API Key → 选择知识点 → 生成 5 题 → 题目格式正确 |
| 完整练习流程 | 选择题库 → 练习 10 题 → 查看成绩 → 错题本有记录 |
| 全真模拟 | 配置整卷 → 答题计时 → 交卷 → 客观题出分 → 主观题 AI 评分完成 |
| 数据备份恢复 | 备份 → 清空数据 → 恢复 → 数据一致 |
| 离线模式 | 断开网络 → 题库练习 / 文档查看 / 已生成试题练习 均可用 |

**手动冒烟测试项（自动化难以覆盖）：**
- 安装包在未安装任何开发工具的干净 Windows 10/11 VM 上正常安装与运行
- SmartScreen 提示行为（记录实际表现，有签名应正常通过）
- 卸载后用户数据目录保留，不被删除

### 10.6 包体积检查

| 组件 | 预期体积 | 超出处理方式 |
|------|----------|-------------|
| Electron 框架 | ~120MB | 剔除多余语言包 |
| Python 服务（无 OCR 语言包） | ~80MB | 排除 pyinstaller 收集的无用包 |
| 前端资源 | ~5MB | 检查是否有未 tree-shake 的大型依赖 |
| 安装包总计（标准版） | ≤ 300MB | 若超出逐一分析大文件 |

### 10.7 验收标准

- [ ] 在干净 VM 上安装成功，不触发 SmartScreen 拦截（或触发后用户明确操作可继续）
- [ ] 所有 Playwright 端到端测试通过
- [ ] 手动冒烟测试无阻断性问题
- [ ] 安装包体积在目标范围内
- [ ] 自动更新在测试环境验证：检测到新版本 → 下载 → 重启 → 版本号更新

## 11. 已知风险与应对

| 风险 | 影响 | 应对方案 | 验证时机 |
|------|------|----------|---------|
| SQLCipher 原生模块在目标 Windows 版本编译失败 | Phase 1 阻断 | 降级为应用层加密敏感字段，整库不加密 | Phase 0 必须验证 |
| PyInstaller 打包后启动时间 > 15 秒 | 用户体验严重下降 | 分析依赖树，剔除未使用大型库；提前预热策略 | Phase 7 |
| Ollama 本地模型内存占用导致系统卡顿 | AI 功能不可用 | 添加内存检测，< 8GB 可用内存时提示风险；限制并发 AI 请求数 | Phase 3 |
| 目标爬虫网站结构改版导致规则失效 | 爬虫功能失效 | 规则文件支持热更新，不随应用打包；提供规则测试工具方便用户自修复 | Phase 5 |
| 代码签名证书申请周期长（EV 证书需 1~2 周审核） | 发布延期 | 提前申请证书；无证书时记录 SmartScreen 已知问题，用标准 OV 证书作临时方案 | Phase 7 前 |
| Tesseract 中文 OCR 准确率不达预期 | PDF 扫描件解析失败 | 优化预处理参数；提供 OCR 结果手动校正界面；记录为已知限制 | Phase 3 |
| chromadb 在低配机器上向量检索性能不足 | 语义检索体验差 | 限制 collection 大小，超出时降级为 FTS5 关键词检索 | Phase 5 |
