# 软考系统架构设计师考试辅助桌面工具 技术方案（方案一：Electron + Python 微服务）

## 1. 文档概述

本文档是《软考系统架构设计师考试辅助桌面工具（单机版）》的技术实现方案，基于 **Electron + Python 微服务** 架构。文档详细定义了技术选型、项目结构、核心模块设计、开发规范、构建与打包流程等，供开发团队实施参考。

### 方案核心思路
- **桌面壳**：Electron 负责窗口管理、系统集成、本地文件访问、数据库读写。
- **前端 UI**：Vue 3 + TypeScript + Naive UI，构建响应式界面。
- **核心业务引擎**：Python 子进程通过本地 HTTP/WebSocket 服务运行，承担 AI 调用、PDF 解析、OCR、爬虫、向量检索等高计算任务。
- **数据存储**：SQLite（Node 端直接操作，WAL 模式，全文检索支持）+ 向量库（Python 端，知识点语义检索）。
- **AI 能力**：Python 端可接本地大模型（Ollama/llama.cpp）或远程 API，密钥由用户在设置中配置，支持纯离线与在线模式切换。

## 2. 整体架构设计

### 2.1 架构逻辑视图

整体应用由 Electron 主进程、Electron 渲染进程、Python 微服务三个主要部分构成。

- **主进程（Main Process）**：负责窗口生命周期、系统托盘、通知、菜单、文件对话框、Python 进程的启动与监控、数据库操作（直接使用 `better-sqlite3` 确保性能）。
- **渲染进程（Renderer Process）**：运行 Vue 应用，通过预加载脚本暴露的安全 API 与主进程通信。所有界面渲染、用户交互均在此层完成。
- **Python 微服务**：独立进程，启动一个仅绑定 `127.0.0.1` 的 FastAPI 服务，接收主进程发来的任务（如生成试卷、评分、解析 PDF、向量检索），执行完毕后返回结果。

### 2.2 进程间通信方式

- **渲染进程 ↔ 主进程**：使用 Electron 的 `contextBridge` 和 `ipcRenderer`，严格限制暴露的 API，杜绝远程模块直接访问。所有 IPC 调用均有超时机制与统一返回格式（见 5.1.4）。
- **主进程 ↔ Python 服务**：通过 `http://127.0.0.1:{随机端口}` 的 REST API 通信，主进程在 spawn Python 时传递端口号及共享密钥。每个请求须携带 `X-Internal-Token` 头，Python 服务验证后才处理，防止本机其他进程非法调用。
- **实时反馈**：对于耗时任务（如 OCR 进度、AI 流式生成），Python 服务通过 WebSocket 推送进度，主进程转发给渲染进程；WebSocket 客户端实现指数退避重连与消息序号校验。

### 2.3 Python 服务启动策略

PyInstaller 打包后冷启动耗时可达 5~15 秒，需明确启动时机以避免首次操作卡顿：

1. **应用启动后后台预热**（推荐）：主进程在主窗口显示后 3 秒开始异步启动 Python 服务，同时前端显示"AI 服务初始化中"状态指示器。
2. **按需懒加载**（备用）：仅在用户首次触发 AI/PDF/OCR 功能时启动，前端展示明确的进度对话框。
3. **空闲回收**：连续 30 分钟无任务请求后自动停止服务，再次使用时重新启动；健康检查间隔内失败超过 3 次则强制重启。

### 2.4 长任务状态机

AI 生成、PDF 解析等耗时任务统一使用以下状态机管理，确保进程崩溃或中途取消时数据一致性：

```
pending → running → completed
                 ↘ failed
                 ↘ cancelled
```

- 任务启动时在数据库 `tasks` 表写入 `running` 状态与启动时间戳。
- 完成时更新为 `completed` 并写入结果；失败或超时更新为 `failed` 并记录错误信息。
- 应用重启时扫描所有 `running` 状态任务，自动标记为 `failed`，前端提示用户重试。
- 渲染进程通过 `AbortController` 机制可发起取消请求，主进程转发取消信号至 Python 端并清理状态。

## 4. 项目结构规范

### 4.1 目录组织

项目根目录下分为三大模块：`electron-app/`（Electron 主项目）、`python-service/`（Python 微服务）、`scripts/`（辅助脚本）。另有 `docs/` 存放文档，`contracts/` 存放双端数据契约定义。

#### electron-app/
- `package.json`：项目依赖与脚本定义。
- `electron.vite.config.ts`：electron-vite 构建配置。
- `src/main/`：主进程代码，包含入口、IPC 处理、数据库操作、Python 进程管理、系统通知、更新器等子模块。
  - `ipc-channels.ts`：**所有 IPC 通道名常量的唯一注册表**，禁止在其他地方硬编码字符串。
- `src/preload/`：预加载脚本，定义暴露给渲染进程的安全 API。
- `src/renderer/`：Vue 前端源码，包含路由、状态管理、组件、视图、API 调用层、静态资源等。
- `resources/`：应用图标等静态资源。

#### python-service/
- `pyproject.toml` / `requirements.txt`：依赖管理。
- `main.py`：FastAPI 服务入口。
- `config.py`：配置管理，包含 PyInstaller 路径适配（`sys._MEIPASS` 兼容）。
- `modules/ai/`：AI 评分、试题生成、知识点抽取、本地与远程模型调用。
- `modules/pdf/`：PDF 文本提取与 OCR 处理。
- `modules/crawler/`：爬虫引擎及规则文件（JSON 格式）。
- `modules/embedding/`：文本向量化、向量库管理、语义检索接口。
- `modules/backup/`：备份打包、加密压缩、WebDAV 上传/下载。
- `modules/utils/`：文本处理、日志等工具。
- `tests/`：Python 单元测试。

#### contracts/
- `*.schema.json`：各接口的 JSON Schema 定义，Python 端 pydantic 模型与 Node 端 zod schema 均从此生成或对齐，确保双端数据契约一致。

### 4.2 命名规范

- **文件命名**：主进程使用 kebab-case（如 `python-bridge.ts`），Vue 组件使用 PascalCase（如 `ExamSimulator.vue`），Python 模块使用 snake_case。
- **IPC 通道名**：统一在 `ipc-channels.ts` 中以常量形式定义，格式为 `namespace:action`，如 `db:queryQuestions`、`ai:generatePaper`，全部小写。

## 5. 核心模块详细设计

### 5.1 Electron 主进程

#### 5.1.1 Python 进程管理

负责 Python 微服务的生命周期管理。

- **启动时机**：应用主窗口加载完成后 3 秒开始后台预热；或在首次调用 AI/PDF 功能时按需启动（前端须显示明确的等待提示）。
- **端口与密钥分配**：动态获取随机可用端口，同时生成一次性 UUID 作为共享密钥（`INTERNAL_TOKEN`），两者均通过环境变量传递给 Python 进程。密钥仅存于内存，不写入磁盘。
- **健康检查**：主进程周期性（每 10 秒）请求 Python 服务的 `/health` 端点；若连续 3 次失败则强制终止并重启进程。
- **优雅关闭**：应用退出时先发送终止信号，等待 5 秒超时后强制结束。
- **日志收集**：Python 进程的标准输出/错误被主进程捕获并记录到日志文件中；开发模式下可转发至渲染进程控制台。

#### 5.1.2 数据库层

使用 `@journeyapps/sqlcipher` 操作加密 SQLite 数据库（原生模块，需通过 `electron-rebuild` 重编译）。

**核心表结构：**

```sql
-- 题目表
CREATE TABLE questions (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  type         TEXT NOT NULL,          -- single/multi/case/essay
  content      TEXT NOT NULL,
  options      TEXT,                   -- JSON，客观题选项
  answer       TEXT NOT NULL,
  explanation  TEXT,
  knowledge_tags TEXT,                 -- JSON 数组，知识点标签
  difficulty   INTEGER DEFAULT 3,      -- 1~5
  source_type  TEXT,                   -- manual/crawled/ai_generated
  source_url   TEXT,
  review_count INTEGER DEFAULT 0,
  last_reviewed_at INTEGER,            -- Unix 时间戳
  created_at   INTEGER NOT NULL
);

-- 做题记录表
CREATE TABLE answer_records (
  id                  INTEGER PRIMARY KEY AUTOINCREMENT,
  question_id         INTEGER REFERENCES questions(id),
  session_id          INTEGER REFERENCES exam_sessions(id),
  user_answer         TEXT,
  is_correct          INTEGER,         -- 客观题 0/1，主观题 null
  time_spent_seconds  INTEGER,
  ai_score            REAL,
  ai_feedback         TEXT,            -- JSON
  manual_override_score REAL,
  created_at          INTEGER NOT NULL
);

-- 学习计划表
CREATE TABLE study_plans (
  id                   INTEGER PRIMARY KEY AUTOINCREMENT,
  exam_date            INTEGER NOT NULL,
  mode                 TEXT DEFAULT 'normal',  -- normal/sprint
  auto_adjust_enabled  INTEGER DEFAULT 1,
  created_at           INTEGER NOT NULL
);

-- 任务状态表（长任务状态机）
CREATE TABLE tasks (
  id          TEXT PRIMARY KEY,        -- UUID
  type        TEXT NOT NULL,           -- ai_generate/pdf_parse/ocr/crawl
  status      TEXT NOT NULL,           -- pending/running/completed/failed/cancelled
  payload     TEXT,                    -- JSON 入参
  result      TEXT,                    -- JSON 结果
  error       TEXT,
  started_at  INTEGER,
  finished_at INTEGER,
  created_at  INTEGER NOT NULL
);

-- 文档表
CREATE TABLE documents (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  title        TEXT NOT NULL,
  file_path    TEXT,
  subject      TEXT,
  source       TEXT,
  page_count   INTEGER,
  imported_at  INTEGER NOT NULL
);

-- 文档段落表（与向量库 ID 对应）
CREATE TABLE doc_chunks (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  document_id   INTEGER REFERENCES documents(id),
  page_number   INTEGER,
  content       TEXT NOT NULL,
  knowledge_tag TEXT,
  vector_id     TEXT                   -- chromadb 中的对应 ID
);
```

**全文检索：** 对 `questions.content` 和 `doc_chunks.content` 建立 FTS5 虚拟表，通过触发器与原始表保持同步。

**数据库迁移：** 使用自定义迁移脚本管理 Schema 版本，每次启动时检查 `user_version` PRAGMA，按序执行未执行的迁移 SQL。

**加密说明：** SQLCipher 密钥由用户设定主密码或首次安装时随机生成，保存在 Windows Credential Manager（通过 `keytar` 访问），不以明文写入任何配置文件。

#### 5.1.3 系统通知与提醒

- 使用 Electron 原生通知接口发送桌面通知。
- 定时任务检查学习计划，计算待完成任务与连续打卡情况，推送相应提醒。
- **番茄钟模式**：学习时段内开启免打扰，计时结束后推送休息提醒，并将专注时长写入 `study_sessions` 表。
- **健康提醒**：通过 `setInterval` 监测单次连续学习时长，超过 45 分钟弹出活动提醒卡片（使用 Electron 窗口悬浮层实现），并记录每日学习时长统计。

#### 5.1.4 IPC 规范

所有 IPC 处理遵循以下约定：

**统一返回格式：**
```typescript
type IpcResponse<T> = 
  | { success: true;  data: T }
  | { success: false; error: { code: string; message: string } }
```

**超时机制：** 渲染进程封装层对每个 IPC 调用设置超时（默认 30 秒，AI 任务 120 秒），超时后 reject 并触发取消流程。

**取消机制：** 长任务启动时返回 `taskId`，渲染进程可通过 `task:cancel` 通道发送取消请求，主进程向 Python 服务发送 DELETE 请求并更新任务状态。

#### 5.1.5 自动更新

使用 `electron-updater` 实现：
- 检查更新服务器（GitHub Releases 或自建 update.json 端点）。
- 静默下载，完成后提示用户选择立即重启或下次启动生效。
- Python 微服务部分无法热更新，需整包替换；更新包中包含新版 Python 可执行文件夹，更新完成后替换 `resources/python-service/` 目录。

### 5.2 Vue 渲染进程

#### 5.2.1 状态管理

使用 Pinia 构建多个 Store：

- `useQuestionStore`：管理题库数据、当前练习状态、筛选条件。
- `useExamStore`：控制模拟考试流程、计时、答题卡。
- `usePlanStore`：学习计划与进度统计，包含自适应调整逻辑触发。
- `useAiStore`：AI 任务队列、进度状态与结果缓存。
- `useAppStore`：全局设置（暗黑模式、AI 配置、通知开关等）。
- `usePomodoroStore`：番茄钟状态、专注时长统计。
- `useAchievementStore`：成就/徽章系统，条件判断与解锁记录。

#### 5.2.2 路由设计

主要路由包括：

- 首页仪表盘（学习概览、倒计时、今日任务）
- 题库练习页及题目详情
- 模拟考试配置与考试会话、成绩报告
- 学习计划管理（含自适应调整与冲刺模式切换）
- 文档库（含全文检索与知识点归类视图）
- 知识图谱可视化页面
- 论文专项训练（含结构引导编辑器与素材库）
- AI 对话页面
- 设置（AI 配置、备份与恢复、通知、健康提醒）

#### 5.2.3 与主进程通信

通过预加载脚本暴露 `window.electronAPI` 对象，渲染进程调用其方法发起 IPC 请求。所有 API 均有 TypeScript 类型定义（与 `ipc-channels.ts` 及 `contracts/` 下的 zod schema 对齐），保证调用安全。

#### 5.2.4 成就系统

在 `useAchievementStore` 中维护成就定义列表与已解锁记录，触发时机包括：
- 连续打卡 N 天
- 累计完成题目数量里程碑
- 某知识点正确率首次超过 90%
- 完成首次全真模拟考试
- 完成首篇论文并获得 AI 评分

成就解锁时通过主进程发送桌面通知，并在界面展示动画弹窗。

### 5.3 Python 微服务

#### 5.3.1 FastAPI 服务定义

微服务启动一个 FastAPI 应用，包含 AI、PDF、爬虫、向量检索、备份五个路由模块，并提供 `/health` 健康检查端点。所有路由通过统一的认证中间件校验 `X-Internal-Token` 请求头，验证失败返回 403。

```python
# config.py：PyInstaller 路径适配
import sys, os

def get_resource_path(relative_path: str) -> str:
    """兼容开发环境与 PyInstaller 打包环境的资源路径解析。"""
    base = getattr(sys, '_MEIPASS', os.path.dirname(os.path.abspath(__file__)))
    return os.path.join(base, relative_path)
```

#### 5.3.2 AI 模块设计

- **统一接口**：定义抽象提供者接口，支持本地推理（Ollama）与远程 API（OpenAI 兼容接口）的切换。
- **结果缓存**：对相同参数的出题/评分请求计算参数哈希，命中缓存直接返回，避免重复调用大模型。
- **出题流程**：
  1. 接收参数（知识点、题型、难度、数量、命题倾向：高频考点/技术趋势等）。
  2. 构建约束性提示词，要求返回结构化 JSON；内置历年高频考点权重配置。
  3. 调用模型生成内容，通过 pydantic 模型解析并校验字段完整性，最多重试 3 次。
  4. 返回题目列表及模型版本信息。
- **评分流程**：
  1. 加载预设的评分标准细项（论文评分维度：摘要完整性、正文深度、项目实践结合度、术语规范性等）。
  2. 将用户答案、参考答案和评分细则组合成提示词。
  3. 要求模型输出 JSON 格式的总分、维度得分与评语。
  4. 结果持久化到 `answer_records` 表，支持人工复核覆盖。

#### 5.3.3 PDF 处理模块

- **文本提取**：优先从 PDF 直接提取文字；若页面文字过少，判断为扫描件并转入 OCR 流程。
- **OCR**：对图像进行预处理（灰度化、二值化、去噪、角度校正）后调用 Tesseract 识别。中文语言包（`chi_sim.traineddata`）约 100MB，不随主程序打包，首次使用 OCR 时提示用户下载并存入用户数据目录。
- **分段与知识点归类**：
  - 基于正则匹配章节标题进行段落合并与层次划分。
  - 使用 `embedding` 模块将段落向量化，与大纲标签进行余弦相似度匹配，实现自动知识点归类，结果写入 `doc_chunks.knowledge_tag`，允许用户通过 UI 手工校正。
- **缓存**：以文件 MD5 为键缓存解析结果，同一文件不重复解析。

#### 5.3.4 向量检索模块

使用 `chromadb` 管理文档段落的向量索引，数据持久化于用户数据目录。

- **索引构建**：PDF 导入完成后异步对每个 `doc_chunks` 条目生成嵌入向量，写入 chromadb collection，同时将 `vector_id` 回写到 SQLite。
- **语义检索**：用户搜索或 AI 问答时，将查询文本向量化后在 chromadb 中检索 Top-K 最相关段落，作为 RAG 上下文。
- **知识图谱**：批量计算段落间相似度矩阵，提取超过阈值的关联对，生成知识点关系图数据，供前端 ECharts 可视化展示。

#### 5.3.5 爬虫引擎

- **抓取模式**：支持基于 CSS/XPath 的规则抓取、手动剪藏 HTML 片段、以及实验性的 AI 辅助解析。
- **规则管理**：每个站点的抓取规则定义为独立的 JSON 配置文件，存储在 `rules/` 目录，应用启动时加载，用户可通过 UI 编辑和即时测试。规则文件支持导入导出，方便分享（不含题库内容）。
- **合规控制**：内置随机延迟（500ms~3000ms 可配置）、User-Agent 池；支持 robots.txt 检查，由用户在设置中开启。

#### 5.3.6 备份与恢复模块

- **备份内容**：SQLite 数据库文件、chromadb 向量库目录、爬虫规则、用户配置。
- **备份格式**：AES-256 加密的 ZIP 压缩包，密码由用户设定或与数据库主密码共用。
- **存储目标**：本地指定目录、可移动存储，或通过 `webdavclient3` 上传至用户配置的 WebDAV 服务（Nextcloud、坚果云等）。
- **定时备份**：支持每日自动备份，最多保留 N 份，超出后删除最旧的。
- **恢复**：从备份文件解密解压，替换当前数据目录，重启后生效。

#### 5.3.7 论文专项训练模块

**结构引导编辑器**（前端为主，Python 辅助）：
- 前端提供分段模板（摘要、背景与挑战、技术方案、项目实践、总结与体会），各段落独立编辑区。
- 每段实时统计字数，AI 可对当前段落给出术语建议和方向提示（通过独立 AI 接口调用）。

**论文素材库**：
- 用户可手动录入工作项目经验（项目背景、技术难点、解决方案），结构化存储。
- 导入的文档中，相关段落可被标记为"可引用素材"。
- 写作时可通过语义检索快速找到相关素材插入引用。

## 6. 开发规范

### 6.1 代码风格

- **前端 (TypeScript/Vue)**：遵循 ESLint 配置与 Prettier 格式化，组件样式使用 scoped 与 BEM 或组件库变量。
- **主进程 (TypeScript)**：遵循 Node.js 最佳实践，所有 IPC 处理器内部必须捕获异常并返回标准 `IpcResponse` 错误格式，避免进程崩溃。
- **Python**：遵循 PEP 8，使用 `black` 格式化、`isort` 排序导入、`mypy` 类型检查；日志使用 `loguru`；所有 API 入参出参均使用 pydantic 模型定义。

### 6.2 数据契约管理

- `contracts/` 目录维护各接口的 JSON Schema，作为 Python pydantic 模型与 Node zod schema 的共同参考源。
- 每次修改接口须同步更新契约文件；CI 中运行契约一致性检查脚本。

### 6.3 Git 工作流

- 采用 `main` → `develop` → `feature/*` 分支策略。
- 提交信息遵循 Conventional Commits 规范：`feat:`, `fix:`, `docs:`, `refactor:` 等。
- 合并到 `develop` 或 `main` 需代码审查，前端/主进程和 Python 至少各一人 Review。

### 6.4 CI 配置

在 GitHub Actions 中配置以下流水线（`.github/workflows/ci.yml`）：

```
push/PR → [并行] ESLint + TypeScript 检查
                Python mypy + pytest
       → [串行] electron-vite 构建验证
              PyInstaller 打包验证（仅 main 分支）
```

发布时额外执行安装包代码签名步骤（详见 7.3）。

### 6.5 错误处理与边界

- **Python 服务不可用**：主进程捕获请求异常，向前端返回统一错误提示，用户可手动重启服务；本地题库练习、文档查看等非 AI 功能不受影响。
- **AI 生成格式异常**：Python 端通过 pydantic 校验，进行最多 3 次重试，若仍失败则返回明确错误信息，前端提示用户检查 AI 配置。
- **大文件处理**：PDF 解析采用分页处理，每批最多处理 10 页；OCR 限制最大并发页数为 4；进度通过 WebSocket 实时推送。
- **数据库事务**：所有多步骤写入操作（如导入 PDF 同时写 documents + doc_chunks）必须包裹在事务中，失败时整体回滚。

## 7. 构建与打包规范

### 7.1 开发环境

- **操作系统**：Windows 10/11，推荐 VS Code 及相应插件。
- **环境准备**：
  1. 安装 Node.js 20 LTS 与 Python 3.11+。
  2. 在 `electron-app` 下执行 `npm install`，随后运行 `electron-rebuild` 重新编译 SQLCipher 原生模块（需安装 Visual Studio Build Tools）。
  3. Python 服务在 `python-service` 目录下创建虚拟环境并安装依赖。
- **启动方式**：
  1. 在 `electron-app` 下启动 Vite 开发服务器（支持热更新）。
  2. Python 服务可单独启动用于调试，也可通过 `scripts/dev-start.ps1` 同时启动两者。

### 7.2 生产构建

1. **Python 打包**：使用 PyInstaller 将 Python 服务打包为独立文件夹，`config.py` 中使用 `sys._MEIPASS` 适配资源路径；不打包 OCR 语言包（按需下载），Tesseract 运行时作为独立资源包含；输出至 Electron 项目的 `resources/python-service/`。
2. **前端与主进程构建**：通过 electron-vite 构建，输出至 `out/` 目录。
3. **Electron 打包**：使用 electron-builder 配置 `extraResources` 包含 Python 分发文件夹及必要依赖，生成 NSIS 安装程序或便携版。

### 7.3 代码签名

未签名的安装包会被 Windows Defender SmartScreen 拦截，**必须进行代码签名**：

- 申请 EV Code Signing 证书（推荐）或标准 OV 证书。
- 在 `electron-builder.yml` 中配置 `win.certificateFile` 与 `win.certificatePassword`（通过环境变量注入，不写入代码仓库）。
- CI 发布流水线中从密钥管理服务获取证书，签名后验证签名有效性再上传 Release。

### 7.4 包体积控制

- 剔除 Electron 多余语言包（仅保留 zh-CN/en-US）。
- OCR 中文语言包不随主程序打包，首次使用时下载。
- 排除 Python 不必要的大型库（如 numpy 的调试符号等）。
- 预期最终安装包：标准版（无 OCR）约 250~300MB，完整版约 350~450MB。

## 9. 安全与隐私设计

- **本地服务隔离**：Python 微服务仅监听 `127.0.0.1`，外部网络无法直接访问；所有请求须携带 `X-Internal-Token` 进行认证，防止本机其他进程非法调用。
- **密钥安全**：用户配置的 AI API 密钥与数据库主密码均通过 `keytar` 存储在 Windows Credential Manager，传递时使用临时环境变量，不以明文写入任何配置文件。
- **学习数据**：SQLite 数据库通过 SQLCipher 加密，存储在应用用户数据目录，密钥管理见 5.1.2。
- **网络行为**：爬虫默认不自动联网，仅在用户手动触发时运行；所有抓取内容仅存储在本地；WebDAV 备份传输使用 HTTPS。
- **合规提示**：应用内明确提示 AI 评分仅供参考；爬虫功能使用前须用户确认合规性，工具不内置任何绕过反爬机制的手段。

## 10. 性能优化指南

- **数据库**：高频写入表开启 WAL 模式，定期执行 `PRAGMA optimize` 与 `VACUUM`。
- **AI 任务**：利用异步 IO 管理并发，同一时间最多允许 2 个 AI 任务并行；长时间生成通过 WebSocket 推送增量结果；相同参数命中缓存直接返回。
- **向量检索**：chromadb 索引在 PDF 导入后台异步构建，不阻塞主流程；检索结果按会话缓存，同一对话内不重复检索。
- **前端渲染**：对题库列表、错题本等长列表采用虚拟滚动技术；知识图谱节点超过 200 个时启用 WebWorker 计算布局。
- **PDF 处理**：以文件 MD5 为键缓存解析结果，同一文件不重复解析；后台分批处理，每批 10 页，批间允许主线程响应其他操作。
