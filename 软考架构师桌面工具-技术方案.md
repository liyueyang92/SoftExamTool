# 软考系统架构设计师考试辅助桌面工具 技术方案（方案一：Electron + Python 微服务）

## 1. 文档概述

本文档是《软考系统架构设计师考试辅助桌面工具（单机版）》的技术实现方案，基于 **Electron + Python 微服务** 架构。文档详细定义了技术选型、项目结构、核心模块设计、开发规范、构建与打包流程等，供开发团队实施参考。

### 方案核心思路
- **桌面壳**：Electron 负责窗口管理、系统集成、本地文件访问、数据库读写。
- **前端 UI**：Vue 3 + TypeScript + Naive UI，构建响应式界面。
- **核心业务引擎**：Python 子进程通过本地 HTTP/WebSocket 服务运行，承担 AI 调用、PDF 解析、OCR、爬虫等高计算任务。
- **数据存储**：SQLite（Node 端直接操作，WAL 模式，全文检索支持）。
- **AI 能力**：Python 端可接本地大模型（Ollama/llama.cpp）或远程 API，密钥由用户在设置中配置，支持纯离线与在线模式切换。

## 2. 整体架构设计

### 2.1 架构逻辑视图

整体应用由 Electron 主进程、Electron 渲染进程、Python 微服务三个主要部分构成。

- **主进程（Main Process）**：负责窗口生命周期、系统托盘、通知、菜单、文件对话框、Python 进程的启动与监控、数据库操作（直接使用 `better-sqlite3` 确保性能）。
- **渲染进程（Renderer Process）**：运行 Vue 应用，通过预加载脚本暴露的安全 API 与主进程通信。所有界面渲染、用户交互均在此层完成。
- **Python 微服务**：独立进程，启动一个仅绑定 `127.0.0.1` 的 FastAPI 服务，接收主进程发来的任务（如生成试卷、评分、解析 PDF），执行完毕后返回结果。仅在需要时启动，空闲时自动关闭以节省资源。

### 2.2 进程间通信方式

- **渲染进程 ↔ 主进程**：使用 Electron 的 `contextBridge` 和 `ipcRenderer`，严格限制暴露的 API，杜绝远程模块直接访问。
- **主进程 ↔ Python 服务**：通过 `http://127.0.0.1:{随机端口}` 的 REST API 通信，主进程在 spawn Python 时传递端口号。支持健康检查，若服务异常则自动重启。
- **实时反馈**：对于耗时任务（如 OCR 进度、AI 流式生成），Python 服务通过 WebSocket 推送进度，主进程转发给渲染进程。

## 4. 项目结构规范

### 4.1 目录组织

项目根目录下分为三大模块：`electron-app/`（Electron 主项目）、`python-service/`（Python 微服务）、`scripts/`（辅助脚本）。另有 `docs/` 存放文档。

#### electron-app/
- `package.json`：项目依赖与脚本定义。
- `electron.vite.config.ts`：electron-vite 构建配置。
- `src/main/`：主进程代码，包含入口、IPC 处理、数据库操作、Python 进程管理、系统通知、更新器等子模块。
- `src/preload/`：预加载脚本，定义暴露给渲染进程的安全 API。
- `src/renderer/`：Vue 前端源码，包含路由、状态管理、组件、视图、API 调用层、静态资源等。
- `resources/`：应用图标等静态资源。

#### python-service/
- `pyproject.toml` / `requirements.txt`：依赖管理。
- `main.py`：FastAPI 服务入口。
- `config.py`：配置管理。
- `modules/ai/`：AI 评分、试题生成、知识点抽取、本地与远程模型调用。
- `modules/pdf/`：PDF 文本提取与 OCR 处理。
- `modules/crawler/`：爬虫引擎及规则文件（JSON 格式）。
- `modules/utils/`：文本处理、日志等工具。
- `tests/`：Python 单元测试。

### 4.2 命名规范

- **文件命名**：主进程使用 kebab-case（如 `python-bridge.ts`），Vue 组件使用 PascalCase（如 `ExamSimulator.vue`），Python 模块使用 snake_case。
- **IPC 通道名**：统一使用 `namespace:action` 格式，如 `db:queryQuestions`、`ai:generatePaper`，全部小写。

## 5. 核心模块详细设计

### 5.1 Electron 主进程

#### 5.1.1 Python 进程管理

负责 Python 微服务的生命周期管理。

- **启动时机**：应用启动后延迟数秒启动，避免影响首屏加载；或在首次调用 AI/PDF 功能时按需启动。
- **端口分配**：动态获取随机可用端口，通过环境变量传递给 Python 进程。
- **健康检查**：主进程周期性请求 Python 服务的健康检查端点；若连续失败多次，则强制终止并重启进程。
- **优雅关闭**：应用退出时先发送终止信号，等待超时后强制结束。
- **日志收集**：Python 进程的标准输出/错误被主进程捕获并记录到日志文件中；开发模式下可转发至渲染进程控制台。

#### 5.1.2 数据库层

使用 `better-sqlite3` 同步 API 操作本地 SQLite 数据库。

- **表设计**：包含题目表、做题记录表、学习计划表、文档表等。题目选项采用 JSON 字段存储，支持知识点标签、难度等维度。
- **全文检索**：对文档表和题目表建立 FTS5 虚拟表，实现高性能模糊搜索，通过触发器保持与原始表的同步。
- **加密**：可选使用 SQLCipher 进行整库加密，密钥由用户设定或随机生成保存在本地安全区域。

#### 5.1.3 系统通知与提醒

- 使用 Electron 原生通知接口或相关社区模块发送桌面通知。
- 定时任务检查学习计划，计算待完成任务与连续打卡情况，推送相应提醒。

### 5.2 Vue 渲染进程

#### 5.2.1 状态管理

使用 Pinia 构建多个 Store：

- `useQuestionStore`：管理题库数据、当前练习状态、筛选条件。
- `useExamStore`：控制模拟考试流程、计时、答题卡。
- `usePlanStore`：学习计划与进度统计。
- `useAiStore`：AI 任务队列与结果缓存。
- `useAppStore`：全局设置（暗黑模式、AI 配置、通知开关等）。

#### 5.2.2 路由设计

主要路由包括：

- 首页仪表盘（学习概览）
- 题库练习页及题目详情
- 模拟考试配置与考试会话、成绩报告
- 学习计划管理
- 文档库
- AI 对话页面
- 设置

#### 5.2.3 与主进程通信

通过预加载脚本暴露 `window.electronAPI` 对象，渲染进程调用其方法发起 IPC 请求。所有 API 均有 TypeScript 类型定义，保证调用安全。

### 5.3 Python 微服务

#### 5.3.1 FastAPI 服务定义

微服务启动一个 FastAPI 应用，包含 AI、PDF、爬虫三个路由模块，并提供一个健康检查端点。服务监听由主进程传入的端口，仅绑定本地回环地址。

#### 5.3.2 AI 模块设计

- **统一接口**：定义抽象提供者接口，支持本地推理（Ollama）与远程 API（OpenAI 兼容接口）的切换。
- **出题流程**：
  1. 接收参数（知识点、题型、难度、数量、命题倾向）。
  2. 构建约束性提示词，要求返回结构化 JSON。
  3. 调用模型生成内容，解析并校验字段完整性。
  4. 返回题目列表及模型版本信息。
- **评分流程**：
  1. 加载预设的评分标准细项（如论文评分维度）。
  2. 将用户答案、参考答案和评分细则组合成提示词。
  3. 要求模型输出 JSON 格式的评分、维度得分与评语。
  4. 返回结构化评分结果。

#### 5.3.3 PDF 处理模块

- **文本提取**：优先从 PDF 直接提取文字；若页面文字过少，判断为扫描件并转入 OCR 流程。
- **OCR**：对图像进行预处理（灰度化、二值化、去噪、角度校正）后调用 Tesseract 识别，提升准确率。
- **分段与知识点归类**：
  - 基于正则匹配章节标题进行段落合并与层次划分。
  - 可选使用本地嵌入模型将段落向量化，与大纲标签进行相似度匹配，实现自动知识点归类，允许用户手工校正。

#### 5.3.4 爬虫引擎

- **抓取模式**：支持基于 CSS/XPath 的规则抓取、手动剪藏 HTML 片段、以及实验性的 AI 辅助解析。
- **规则管理**：每个站点的抓取规则定义为独立的 JSON 配置文件，存储在 `rules/` 目录，应用启动时加载，用户可通过 UI 编辑和即时测试。
- **合规控制**：内置随机延迟、User-Agent 池；支持 robots.txt 检查，由用户在设置中开启。

## 6. 开发规范

### 6.1 代码风格

- **前端 (TypeScript/Vue)**：遵循 ESLint 配置与 Prettier 格式化，组件样式使用 scoped 与 BEM 或组件库变量。
- **主进程 (TypeScript)**：遵循 Node.js 最佳实践，所有 IPC 处理器内部必须捕获异常并返回标准错误格式，避免进程崩溃。
- **Python**：遵循 PEP 8，使用 `black` 格式化、`isort` 排序导入、`mypy` 类型检查；日志使用 `loguru`。

### 6.2 Git 工作流

- 采用 `main` → `develop` → `feature/*` 分支策略。
- 提交信息遵循 Conventional Commits 规范：`feat:`, `fix:`, `docs:`, `refactor:` 等。
- 合并到 `develop` 或 `main` 需代码审查，前端/主进程和 Python 至少各一人 Review。

### 6.3 错误处理与边界

- **Python 服务不可用**：主进程捕获请求异常，向前端返回统一错误提示，用户可手动重启服务。
- **AI 生成格式异常**：Python 端进行有限次重试，若仍失败则返回明确错误信息。
- **大文件处理**：PDF 解析采用分页处理，限制内存占用；OCR 限制最大并发页数。

## 7. 构建与打包规范

### 7.1 开发环境

- **操作系统**：Windows 10/11，推荐 VS Code 及相应插件。
- **启动方式**：
  1. 在 `electron-app` 下安装依赖并启动 Vite 开发服务器（支持热更新）。
  2. Python 服务可单独启动用于调试，也可通过脚本同时启动。

### 7.2 生产构建

1. **Python 打包**：使用 PyInstaller 将 Python 服务打包为独立文件夹或可执行文件，包含 OCR 语言包和 Tesseract 运行时，输出至 Electron 项目的资源目录。
2. **前端与主进程构建**：通过 electron-vite 构建，输出至 `out/` 目录。
3. **Electron 打包**：使用 electron-builder 配置 `extraResources` 包含 Python 分发文件夹及必要依赖，生成 NSIS 安装程序或便携版。
4. **包体积控制**：剔除 Electron 多余语言包，排除 Python 不必要的大型库，预期最终安装包 350~450MB。

## 9. 安全与隐私设计

- **本地服务隔离**：Python 微服务仅监听 `127.0.0.1`，外部网络无法直接访问。
- **密钥安全**：用户配置的 API 密钥通过 Electron 的安全存储 API 加密保存，传递时使用临时环境变量。
- **学习数据**：SQLite 数据库默认存储在应用用户数据目录，可选用 SQLCipher 加密，由用户设定主密码。
- **网络行为**：爬虫默认不自动联网，仅在用户手动触发时运行；所有抓取内容仅存储在本地。

## 10. 性能优化指南

- **数据库**：高频写入表开启 WAL 模式，定期执行优化操作。
- **AI 任务**：利用异步 IO 管理并发，长时间生成通过 WebSocket 推送增量结果。
- **前端渲染**：对长列表采用虚拟滚动技术。
- **PDF 处理**：后台线程处理并缓存结果，避免重复解析。
